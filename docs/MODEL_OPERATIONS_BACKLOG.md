## 目的

Kompari で複数の公式 AI が予測を競う仕組みを、モデル改廃や一時障害に対して壊れにくく、かつ利用者を誤認させない形で運用するための方針をまとめる。あわせて、レース後の「AI 反省会」を将来の学習改善(Prediction Context Pack / AI 別 Learning Memory)へつなぐための土台を定義する。

本文書は「思想」を確定させるものであり、実装の詳細仕様は各項目を PR 化する段階でレビューして決める。MVP スコープを不用意に広げないため、「今すぐ実装」と「将来」を明確に分ける。

---

## 背景

初回の実データ投入(笠松1R C13組)で、Gemini の旧モデル ID `gemini-2.5-flash` / `gemini-2.5-pro` が提供終了(Google の generateContent が明示的に 404 "no longer available" を返す)しており、Gemini だけ実 AI 応答が得られず mock 予測が保存・表示される問題が起きた。

切り分けの結果:

- キー・課金・認証は正常。モデル ID の陳腐化が原因だった。
- 代替候補 `gemini-3.5-flash` は複数回の疎通テストで 503(継続的高負荷)が続き、本番運用に不安があった。
- 実際に疎通が通った `gemini-3.1-flash-lite`(非 preview・固定 ID・models.list 実在確認済み)へ更新し、5AI すべて `isMock:false` の実予測を揃えた。

この一件は偶発事故ではなく、モデル管理が全 AI 手動ハードコードで、本番投入して初めて壊れに気づく構造の表面化である。以下の方針はその再発防止と、利用者の信頼を損なわないための整理である。本 v1.1 は初版に対する設計レビュー(実装時に事故になる箇所の指摘)を反映したものである。

## 設計原則

Kompari の既存の信頼原則に、本件で得た教訓を接続する。

**mock は「作らない → 保存しない → 見せない」の順で断つ。** mock(システムが生成した偽の予測)への対策は、上流から順に、(1)生成しない(mock 許容の生成経路を残さない)、(2)保存しない(Firestore に mock を永続化しない)、(3)見せない(既存データや例外で残った mock を公開画面で確実に除外する)、の三段構えとする。本質は「作らない」であり、表示除外は最後の防衛線にすぎない。予測を取得できなかった AI は「予測した」ではなく「不参加(システム不具合)」として、事実を正しく表示する。これは confidence 表示の削除・outcome 廃止・未実装カテゴリの非表示・sample 除外と同じ系譜にある。

**どのモデルが予測したかを記録する。** 「最新かどうか」より「実際にどのモデルが応答し、その成績がどうか」を追跡できることが Kompari の価値である。リトライやフォールバックでモデルが切り替わった場合も、実際に使ったモデル名を必ず記録し、その名前で成績を集計する。ブランド名(例: "Gemini")で複数モデルの成績を丸めない。

**派生値は保存せず、導出する。** 予測と結果から毎回計算できる値(hit/miss、Top3 系指標など)は、原則として予測 doc に保存しない。保存すると result 訂正時の再計算漏れと SoT の二重化を招く。これは outcome フィールドを廃止したのと同じ理由であり、同型のフィールドを名前を変えて再導入しない。集計コスト等でスナップショット保存が必要になった場合は、その時点で明示的に設計判断する。

**予測は結果確定後に不変。** 結果確定後に予測 doc を書き換えない(既存の UI・関数レベルのガードを維持する)。反省会など結果確定後に生成されるデータは、予測 doc に追記せず、別の保存先に置く。

**最新より、実疎通済み・非 preview・固定 ID・安定を優先する。** 利用者納得感の観点では最新モデルが望ましいが、提供終了リスクの高い preview 版や、中身が予告なく変わる latest エイリアスは、成績の再現性と安定性を損なう。採用は generateContent での疎通確認を通ったものに限る。モデル名を推測してコードに入れない(models.list に載っていても generateContent が 404/503 を返す事例が実在する)。

**恒久エラーと一時エラーを区別する。** 404(モデル廃止)・400(不正リクエスト)・401/403(認証)は何回試しても直らないため、リトライせず原因別に扱う。503/429/timeout/5xx はリトライで直りうる一時障害として扱う。

**モデル切替は自動検知＋人間承認。** モデルの完全自動切替はしない。壊れたことを自動で検知し、切替は人間が承認する。運用者が把握しないまま予測モデルが入れ替わる状態を避ける。

## データ管理方針

予測・モデル・分析データの扱いを定義する。詳細なフィールド名・スキーマは実装 PR 時に確定するが、思想として以下を守る。

**予測に記録すべきモデル情報。** 各予測 doc には、実際に呼び出したモデルを一意に特定できる情報を保存する。フォールバックが発生した場合は、それが分かる形で残す。想定フィールド(実装時に確定):

- `modelName` / `modelId`: 実際に応答したモデルの表示名と ID
- `fallbackFrom`: 本来使う予定だったが失敗したモデル(フォールバック時のみ)
- `fallbackReason`: フォールバックの理由(例: 404 / 503 継続 / timeout)
- `attemptCount`: 成功までに要した試行回数

成績集計は `modelName` / `modelId` 単位で行う。「Gemini というブランドの中でモデルが切り替わった」ことが後から追跡できる状態を保つ。なお、これらは「生成時の事実の記録」であり、結果から導出できる派生値ではないため保存対象である(派生値の保存禁止とは矛盾しない)。

**mock の扱い。** 生成経路から mock を排除する(strict 生成に一本化)。既存データや例外経路で mock が残る場合に備え、表示層でも `isMock:true` を公開画面(イベント詳細・コンセンサス・ランキング・AI 詳細・ホーム等)から確実に除外する。予測を取得できなかった AI 枠には「システム不具合により不参加」の趣旨を表示する。

**「不参加」と「未生成」の区別(実装時の設計論点)。** strict 生成は失敗時に何も保存しないため、現状 Firestore 上では「生成に失敗した(不参加)」と「まだ生成していない(未生成)」を区別できない。「不参加」表示を実装するには、軽量な失敗マーカーを残すか、期待 AI 集合(`OFFICIAL_AI_NAMES`)と既存予測の差分＋イベント状態から推定するか、いずれかのデータソース設計が必要である。この設計は A 項目の read-only 調査の必須論点とする(今すぐスキーマを追加する必要はない)。

**的中判定と分析指標の分離。** 公式の的中率は本命(main)と結果(winner)の一致でのみ判定する(既存の `prediction.main === result.winner` 動的判定を維持)。二番手(second)・三番手(third)は公式的中率には混ぜない。`winnerInTop3` / `winnerRank` / `top3Overlap` / `rankScore` などの分析指標は、**予測 doc に保存せず、集計・表示時に動的算出する**(派生値の保存禁止原則に従う)。利用者に見せる場合も、公式的中率とは別枠であることを明示し、混同させない。

**レース後 AI 反省会(将来)。** レース確定後、各 AI に「なぜ勝ち馬が勝ったか」「自分は何を見落としたか」「次回重視すべき要素は何か」を振り返らせ、保存する。反省会の内容は事実ではなく仮説として扱う。**保存先は予測 doc とは別のコレクション(または別 doc)とし、予測 doc には一切追記しない**(予測不変原則)。この仮説を、AI 別 Learning Memory や Prediction Context Pack の改善材料に用いる。反省会の出力を公式成績や公開の断定的表示に流用しない(仮説を事実として見せない)。

**モデル監査ログ(運用ログ)。** モデルの調査・疎通・採用・廃止の履歴を運用ログとして残す。今回の Gemini の例では「2.5 系 → 404(廃止確定) / 3.5-flash → 503×3(見送り) / 3.1-flash-lite → 疎通成功(採用)」という履歴である。この知識は Gemini 固有ではなく、OpenAI・Anthropic・DeepSeek・xAI でも将来必ず起きる。Learning Memory(AI の予測改善)とは別物の、運用者のためのログである。**実装はまず `docs/MODEL_AUDIT_LOG.md` 等の文書ログで開始してよい(コード不要)**。構造化(Firestore 等)は必要になった時点で判断する。

## 優先順位(v1.1)

### 優先度 A — 今すぐ(MVP 品質・原則直結)

1. **[完了 2026-07-11] mock を作らない。** `/admin/edit` の「全 AI 再生成」ボタン(mock 許容経路)を strict 化または撤去し、mock を生成しうる最後の経路を断つ。mock 対策の本丸。→ `generateAllPredictions` に `allowMock:false` を追加し strict 化(一括ボタンは維持)。AI 別ボタンと合わせて、管理画面の通常運用経路から mock 許容呼び出しが無くなった。詳細: `docs/MIGRATION_STATUS.md`「全AI再生成をstrict化(mock「作らない」P1完了)」。次は項目2(見せない)。
2. **[P2-1完了 2026-07-11 / P2-2は別PR] mock を見せない。** 公開画面の表示ロジックで `isMock:true` を確実に除外する(既存データ・例外経路への最後の防衛線)。→ イベント詳細(`app/events/[slug]/page.tsx`)の公開表示と表示側集計(カード列挙・件数・コンセンサス分母・podium・優勢/全会一致ラベル・split meter)を、正式予測集合へ統一した。正式性のSoTは `predictionSource === "official-ai"` とするホワイトリスト方式(`lib/events.ts` の `isOfficialPrediction` 1関数に集約)。`official-ai` を主軸としつつ、矛盾データ(`isMock:true` が誤って残ったもの)・My AI(`source==="user"`/`myAiId`)・非公式AI名・main空も多層防御として除外する。`predictionSource` 未設定の旧データ(mock・非公式問わず)は表示対象外になるが、製品方針上これらは正式な実績・ランキング・学習資産として後方互換を保証する対象ではないため、非表示になっても問題ないと判断した。予測が存在しないAIの枠は表示しない(存在する正式予測だけを列挙する元の形を維持)。5AI固定枠・「予測なし」表示は今回実装しない。正確な「不参加」表示は、イベント作成時点の期待AI集合(`expectedAiNames`)を保存する設計が整ってから、項目3(不参加設計)と合わせて別タスクで検討する。ホーム・一覧(P2-2)は未対応、別PRで対応する。
3. **不参加設計。** 「不参加(生成失敗)」と「未生成」を区別するデータソース(軽量な失敗マーカー or 期待 AI 集合との差分推定)を read-only 調査で設計し、「システム不具合により不参加」表示につなぐ。今すぐのスキーマ追加は不要、設計の確定が A の範囲。
4. **モデル疎通ヘルスチェック。** 各 AI の設定モデルが generateContent で応答するかを軽量に確認し、失敗を運用者に可視化する。「本番投入して初めて壊れに気づく」を無くす。実 API コストがかかるため頻度を絞る(例: 生成セッション開始時に 1 回、または日次)。

### 優先度 B — イベント自動化前(可用性・運用性)

5. **エラー種別分岐リトライ。** 503/429/timeout/5xx は最大 3 回リトライ(指数バックオフ)。404/400/401/403 はリトライせず原因別に扱う。手動運用中は「strict 失敗 → alert → 管理者が再クリック」が実質リトライとして機能しているため、自動リトライはイベント自動化の前提整備と位置づける。
6. **Gemini 格上げの定期再検証。** 落ち着いたタイミングで `gemini-3.5-flash` や pro 級を再疎通し、安定して通るなら採用。「安定して 200 を返す」が絶対条件。
7. **モデル監査ログの運用。** まず `docs/MODEL_AUDIT_LOG.md` の文書ログとして開始(今回の Gemini 履歴を初エントリとする)。他 4 AI の障害・切替時にも同形式で追記。構造化は必要になってから。
8. **段階フォールバック。** 本命モデルが恒久エラーの場合に一段下のモデルへフォールバックし可用性を上げる。ただし `modelName` / `fallbackFrom` / `fallbackReason` / `attemptCount` の記録と、その名前での成績集計が絶対条件。記録なしでブランド名に丸めるフォールバックは禁止。

### 優先度 C — 将来(学習基盤・スコープ判断が要る)

9. **Top3 分析指標の可視化。** `winnerInTop3` / `winnerRank` / `top3Overlap` / `rankScore` を集計・表示時に動的算出し、公式的中率とは別枠の分析データとして扱う(保存しない)。
10. **レース後 AI 反省会。** 予測 doc とは別コレクションに保存し、仮説として扱う。予測 doc には一切追記しない。
11. **Learning Memory / Prediction Context Pack への接続。** 反省会の仮説を AI 別 Learning Memory と Prediction Context Pack の改善ループに用いる。中核ループ(予測→結果→成績)が実データで安定してから着手する。
12. **モデルレジストリの外部化。** モデル ID をコードから設定(環境変数 / Firestore / 管理画面)へ移し、コード変更・再デプロイなしで切替可能にする。切替は自動検知＋人間承認。
13. **latest エイリアス戦略。** `*-latest` 系は廃止 404 に強い一方、中身が変わり成績の再現性を損なう。採用するなら「呼び出しは latest、返ってきた実モデルバージョンを記録」する折衷設計を前提に検討する。

## 今後の実装バックログ(着手時の進め方)

各項目は着手時に、まず read-only 調査で現状の生成・表示・エラーハンドリングロジックを正確に把握し、そのうえで小さい PR に分割する。「思想は SoT、実装設計はレビュー資産」の原則に従い、本文書の思想を実装が満たしているかを検証してから確定する。

A の推奨着手順: **作らない(A-1) → 見せない(A-2) → 不参加設計(A-3) → ヘルスチェック(A-4)**。A-1〜A-3 は同一の read-only 調査(mock 生成経路・表示箇所・不参加のデータソース候補の洗い出し)でまとめて把握し、PR は分割する。

いずれも「今すぐの中核ループ運用(実データでの予測→結果→成績)」とは別軸であり、中核ループが実データで一周・安定してから改善フェーズとして着手する。

## スコープに関する注記

本文書は方針の確定であり、実装範囲の確定ではない。優先度 A の 4 項目のみを直近の実装候補とし、B・C は将来項目として据え置く。「予測は結果に先立つ」「実体のないものを見せない」「どのモデルが予測したかを記録する」「派生値は保存せず導出する」「予測は結果確定後に不変」という Kompari の中核原則に反しない範囲でのみ、各項目を実装に落とす。反省会・Learning Memory・Prediction Context Pack といった学習基盤は、MVP(競馬中心・手動運用・5 公式 AI の予測比較メディア)のスコープを広げないよう、中核ループの安定後に扱う。
